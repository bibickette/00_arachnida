YCBCR
YCbCr est un modèle de couleur utilisé en vidéo et en traitement d’image numérique, qui sépare l’information de luminance (Y') de celle des composantes de chrominance (Cb et Cr).  La sous-échantillonnage YCbCr consiste à réduire la résolution des composantes de couleur (chrominance) par rapport à la luminance, exploitant le fait que l’œil humain est moins sensible aux variations de couleur qu’aux variations de luminosité. 

Ce processus permet de réduire la taille des fichiers et la bande passante nécessaire pour transmettre ou stocker des vidéos, sans une perte significative de qualité perçue. Les ratios de sous-échantillonnage, comme 4:2:0, 4:2:2, 4:1:1 ou 4:4:4, indiquent la manière dont les échantillons de chrominance sont échantillonnés par rapport à la luminance dans une région de référence de 4 pixels de large et 2 pixels de haut. 

4:4:4 : Aucune sous-échantillonnage ; chaque pixel a ses propres valeurs Cb et Cr.  Utilisé en post-production haute qualité.
4:2:2 : Les composantes Cb et Cr sont échantillonnées à la moitié de la résolution horizontale, mais pleine résolution verticale.  Courant dans les formats professionnels comme Digital Betacam ou ProRes.
4:2:0 : Les composantes Cb et Cr sont échantillonnées à la moitié de la résolution horizontale et verticale.  Très courant dans les formats de diffusion (DVD, Blu-ray, H.264, AVCHD).
4:1:1 : La résolution horizontale de la chrominance est réduite à un quart.  Utilisé dans des formats comme DV.
Le sous-échantillonnage est une technique essentielle dans les codecs vidéo modernes, permettant un équilibre optimal entre qualité visuelle et efficacité de stockage



EXIF BYTE ORDER
Exif byte order refers to the way multi-byte data (like integers) is stored in an Exif file, determined by the first two bytes of the TIFF header within the Exif data. 

II (Intel, Little-endian): The least significant byte comes first. For example, the 32-bit value 0x12345678 is stored as 78 56 34 12. 
MM (Motorola, Big-endian): The most significant byte comes first. The same value 0x12345678 is stored as 12 34 56 78. 
This byte order is critical for correctly interpreting all subsequent data in the Exif metadata, including tag values, offsets, and counts. Most digital cameras use little-endian (II), while some, like certain Ricoh models and older Sony cameras, use big-endian (MM).

Yes, there are only two possible byte orders in ExIF: little-endian (II) and big-endian (MM). 

The Exif standard, based on the TIFF format, uses a 2-byte indicator at the start of the Exif data to specify the byte order:

"II" (0x4949): Indicates little-endian (Intel) byte order. 
"MM" (0x4D4D): Indicates big-endian (Motorola) byte order.
These are the only two values defined and recognized by the Exif specification for byte order. No other byte orders are supported.

                @classmethod ça veut dire quoi ?

Ça veut dire :

👉 la méthode reçoit la classe elle-même en premier argument.

Par convention on l’appelle :

cls
✅ Comparaison claire
🔵 Méthode normale (instance)
class A:
    def foo(self):
        print(self)

Quand tu fais :

a = A()
a.foo()

👉 Python passe automatiquement :

self = a   # l'objet
🟢 Classmethod
class A:
    @classmethod
    def foo(cls):
        print(cls)

Quand tu fais :

A.foo()

👉 Python passe automatiquement :

cls = A   # la classe


Type	reçoit	sert à
instance method	self	agir sur un objet
classmethod	cls	agir sur la classe
staticmethod	rien	helper logique

Oui, juste après la signature GIF89a, tu as le Logical Screen Descriptor (LSD) qui fait 7 bytes, et il commence bien par Width et Height.

Ta séquence :

b'&\x01\x80\x00\xf6\x00\x00' (7 bytes)

Décomposition (spéc GIF) :

Canvas Width (2 bytes, little-endian)
Canvas Height (2 bytes, little-endian)
Packed Fields (1 byte)
Background Color Index (1 byte)
Pixel Aspect Ratio (1 byte)
Donc ici :

1) Width / Height
Les GIF utilisent du little-endian (contrairement à JPEG souvent).

Width = bytes 26 01 (c’est 0x0126) → 294
Height = bytes 80 00 (c’est 0x0080) → 128
Tu peux le décoder comme ça :

gif_lsd_decode.py
import struct

lsd = b'&\x01\x80\x00\xf6\x00\x00'
width, height, packed, bg, aspect = struct.unpack("<HHBBB", lsd)

print(width, height)      # 294 128
2) C’est quoi le 0x80 ?
Dans ton dump, 0x80 est le premier octet de la hauteur (height low byte), pas un champ séparé.

Height = 0x0080 = 128, donc tu vois \x80\x00.

3) Le byte 0xF6 (packed fields), c’est quoi ?
0xF6 = 11110110 en binaire. Dans le LSD, ça se décompose :

bit 7 : Global Color Table Flag (GCT)
1 → oui, il y a une palette globale après le LSD
bits 6–4 : Color Resolution (valeur + 1)
ici 111 = 7 → “8 bits” (souvent présenté comme “depth 8”)
bit 3 : Sort Flag
ici 0 → non triée (souvent ignoré)
bits 2–0 : Size of Global Color Table (valeur + 1, puissance de 2)
ici 110 = 6 → taille = 2^(6+1)= 128 couleurs dans la palette globale
Donc ce 0xF6 te dit notamment : palette globale de 128 couleurs.

Tu peux décoder ça :

gif_packed_decode.py
packed = 0xF6
gct_flag = (packed >> 7) & 1
color_resolution = ((packed >> 4) & 0b111) + 1
sort_flag = (packed >> 3) & 1
gct_size = 2 ** (((packed & 0b111)) + 1)

4) Background / Aspect
Background Color Index = 0x00 → index 0 dans la palette
Pixel Aspect Ratio = 0x00 → “non spécifié” (cas le plus courant)
Si tu veux, donne-moi les bytes suivants juste après ces 7 bytes : si GCT flag = 1, les octets suivants doivent être la Global Color Table de taille 3 * gct_size (= 384 bytes ici), puis ensuite arrivent les blocs (Graphic Control Extension, Image Descriptor, etc.).
