task_done() — à quoi ça sert ? 🔍

Le problème qu'il résout

Ce que task_done() signifie
C'est toi qui dis manuellement à la Queue : "j'ai fini de traiter cet élément".

task_done() seul ne sert à rien. Il existe uniquement pour fonctionner avec queue.join().
queue.join() bloque le programme principal jusqu'à ce que unfinished_tasks == 0.

Sans ce mécanisme, ton programme principal ne saurait jamais quand tous les workers ont terminé leur travail.

## 3. Pourquoi `queue.join()` et pourquoi DANS le `with`

**Pourquoi `queue.join()`** — le `with ThreadPoolExecutor` attend que les workers **terminent leur fonction**. Mais tes workers sont des boucles `while True` — elles ne terminent jamais d'elles-mêmes. Le `with` attendrait indéfiniment.

`queue.join()` lui, attend que tout le **travail** soit terminé — c'est une attente différente :
```
executor (with)  attend que les fonctions worker() se terminent
queue.join()     attend que tous les items soient traités (unfinished_tasks == 0)
Pourquoi DANS le with — si tu le mets dehors :
pythonwith ThreadPoolExecutor(max_workers=10) as executor:
    executor.submit(self.worker)
# ← ici le with ferme → attend que les workers finissent
# ← workers ont une boucle infinie → FREEZE
# ← on n'atteint jamais queue.join()

self.queue.join()  # ← jamais atteint
À l'intérieur :
pythonwith ThreadPoolExecutor(max_workers=10) as executor:
    executor.submit(self.worker)
    self.queue.join()  # ← BLOQUE ici jusqu'à unfinished_tasks == 0
                       # workers encore vivants pendant ce temps
# ← queue.join() débloqué → on sort du with → workers reçoivent le signal d'arrêt
```

## 4. Pourquoi `task_done()`

La Queue maintient un compteur interne `unfinished_tasks` :
```
queue.put()       → unfinished_tasks + 1   ("j'ai ajouté du travail")
queue.get()       → rien ne change         ("j'ai pris le travail, mais pas fini")
queue.task_done() → unfinished_tasks - 1   ("j'ai FINI ce travail")
queue.join() bloque jusqu'à unfinished_tasks == 0. Sans task_done(), ce compteur n'atteint jamais 0 — freeze infini.
La règle absolue : un get() = un task_done(), toujours dans un finally :
pythonurl, depth = self.queue.get()   # unfinished_tasks + 1
try:
    # traitement...
finally:
    self.queue.task_done()       # unfinished_tasks - 1, quoi qu'il arrive

5. Le vidage de queue et get_nowait()
pythonwhile not self.queue.empty():
    try:
        self.queue.get_nowait()
        self.queue.task_done()
    except Empty:
        break
```

`get_nowait()` c'est `queue.get()` sans attente — si la queue est vide, il lève `Empty` immédiatement au lieu de bloquer.

**Est-ce qu'il "vole" les URLs déjà prises par `queue.get()` ?** Non. Une fois qu'un worker a fait `queue.get()`, l'item est **retiré** de la queue — il n'est plus accessible à personne. `get_nowait()` ne peut prendre que ce qui est encore dans la queue.
```
Queue : [url1, url2, url3, url4]

Worker 1 → get() → prend url1  (url1 disparaît de la queue)
Worker 2 → get() → prend url2  (url2 disparaît de la queue)

Queue maintenant : [url3, url4]

get_nowait() → prend url3 → task_done()
get_nowait() → prend url4 → task_done()
get_nowait() → Empty → break
```

Pourquoi on ne peut pas juste faire `queue.clear()` ? Parce que ça n'existe pas en Python — le contrat `get()/task_done()` doit toujours être respecté, sinon `queue.join()` a un compteur faux.

## 6. Les threads sont-ils vraiment en même temps ?

**Oui et non** — et c'est là que Python est particulier.

Python a le **GIL** (Global Interpreter Lock) — un verrou global qui dit qu'**un seul thread Python peut exécuter du bytecode à la fois** :
```
Thread 1  ██░░░░██░░░░██    ← exécute du code Python
Thread 2  ░░██░░░░██░░░░    ← attend le GIL
Thread 3  ░░░░██░░░░██░░    ← attend le GIL
```

Ça semble nul, mais pour ton scraper **ça ne change rien**. Pourquoi ? Parce que `requests.get()` passe la majorité du temps à **attendre la réponse réseau** — pendant ce temps, le thread relâche le GIL automatiquement :
```
Thread 1  ██[attente réseau....................]██
Thread 2  ░░░░██[attente réseau................]░░██
Thread 3  ░░░░░░██[attente réseau..............]░░░░██
           ↑ pendant l'attente réseau, les autres peuvent tourner
```

Les threads s'exécutent vraiment en parallèle pendant les I/O (réseau, disque). C'est exactement ton cas — ton scraper est **I/O bound**, donc le GIL ne te pénalise pas.

## 7. Python synchrone ou asynchrone ? Comparaison avec C

Le C est **synchrone par défaut** — chaque ligne s'exécute dans l'ordre, une après l'autre. Pour faire du parallèle en C, tu crées des threads POSIX (`pthread_create`) manuellement — tout est bas niveau.

Python est aussi **synchrone par défaut**, mais avec plusieurs façons de faire du parallèle :
```
Synchrone (défaut)      ligne 1 → ligne 2 → ligne 3 → ...
                        tout est séquentiel, prévisible

Threads (ton cas)       plusieurs fils d'exécution simultanés
                        partagent la mémoire → nécessite des locks
                        GIL : un seul thread Python actif à la fois
                        MAIS relâché pendant I/O → vrai parallélisme réseau

Asyncio (autre option)  UN seul thread, mais il "jongle" entre les tâches
                        pendant qu'une tâche attend le réseau, il fait autre chose
                        pas de GIL, pas de locks nécessaires
                        plus complexe à écrire (async/await partout)
Pour du scraping, les deux (threads et asyncio) fonctionnent bien. Tu as choisi les threads — c'est un choix valide et plus simple à comprendre pour commencer.

## La règle simple
```
Tu as lu response.text        → pas besoin de close()
Tu as lu iter_content()       → pas besoin de close()
Tu n'as rien lu du tout       → BESOIN de close()

response.close() sur une réponse déjà entièrement lue est sans effet — requests détecte que le body a été consommé et la connexion est déjà retournée au pool. Tu ne peux pas double-fermer accidentellement. Donc appeler close() dans tous les cas dans le finally est la façon la plus sûre et la plus simple.
